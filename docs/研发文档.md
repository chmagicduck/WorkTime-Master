# 研发文档 - 工时管理大师

## 一、设置数据存储模型

### 1.1 新增 / 调整 Storage Keys

| 字段名 | 类型 | 说明 | 默认值 |
|--------|------|------|--------|
| `MONTHLY_SALARY` | number | 月薪（税前，元） | 无（必填）|
| `LUNCH_BREAK_ENABLED` | boolean | 午休抵扣开关，开启后从有效工时中扣除午休时长 | false |
| `LUNCH_START_HOUR` | number | 午休开始时（0–23） | 12 |
| `LUNCH_START_MIN` | number | 午休开始分（0–59） | 0 |
| `LUNCH_END_HOUR` | number | 午休结束时（0–23） | 13 |
| `LUNCH_END_MIN` | number | 午休结束分（0–59） | 0 |
| `WORK_MODE` | string | 工作日模式，见 1.2 | 'double_rest' |
| `ALTERNATE_WEEK_REF_DATE` | string | 大小周参考周的周一日期（YYYY-MM-DD） | 无 |
| `ALTERNATE_WEEK_IS_BIG` | boolean | 参考日期所在周是否为大周（双休） | 无 |
| `ALTERNATE_WEEK_REST_DAY` | string | 大小周模式小周的休息日：'sat' 或 'sun' | 无 |
| `ONBOARDING_DONE` | boolean | 是否已完成引导配置 | false |
| `WEEKLY_SESSION_RECORDS` | SessionRecord[] | 近 7 天的工作状态分段记录（支持时间轴 & 手动编辑），见 1.5 | [] |

### 1.2 WORK_MODE 取值说明

| 值 | 含义 |
|----|------|
| `double_rest` | 双休：周六、周日均休 |
| `sat_only` | 仅周六休 |
| `sun_only` | 仅周日休 |
| `alternate` | 大小周：双休与单休交替，单休日由 `ALTERNATE_WEEK_REST_DAY` 指定 |

### 1.3 原有 Key 变更说明

- **`HOURLY_WAGE`**：不再由用户输入，改为保存配置时由系统自动计算并存储（见 1.4），其余模块仍直接读取此 Key
- **`REST_DAY_TYPE`**：废弃，由 `WORK_MODE` 替代

### 1.4 时薪自动计算逻辑（保存设置时执行）

时薪 = 月薪 ÷（月均工作日 × 日工作小时数）

**月均工作日：**

| 工作模式 | 月均工作日 |
|----------|-----------|
| 双休 | 21.75 天 |
| 仅周六休 / 仅周日休 | 26 天 |
| 大小周 | 23.875 天（双休和单休各半的平均值）|

**日工作小时数：**（下班时间 - 上班时间）的分钟数 ÷ 60，再减去午休时长（若开启午休）

示例：9:00–18:30，午休 12:00–13:00 → 日工作 8.5 小时

### 1.5 SessionRecord 数据结构（近 7 天分段记录）

```
SessionRecord {
  id: string          // 唯一 ID（时间戳字符串）
  date: string        // YYYY-MM-DD（所属日期）
  type: 'work' | 'fish' | 'overtime'
  startTs: number     // 段开始时间戳（毫秒）
  endTs: number       // 段结束时间戳（毫秒，0 表示当前段仍在进行中）
  earnings: number    // 本段收益（overtime 段固定为 0）
}
```

用途：
- 首页 24 小时活动轨迹热力图的数据源
- 支持未来「手动新增 / 编辑 / 删除 7 天内摸鱼和加班记录」功能

## 二、全局数据清除逻辑

### 2.1 触发入口
个人页「清除全局数据」按钮 → 二次确认底部弹窗 → 点击「确定删除」后执行

### 2.2 清除范围
调用 `wx.clearStorageSync()` 清除**全部** Storage，包括：
- 用户配置（月薪、时薪、工作时间、发薪日、工作模式、午休设置等）
- 引导状态（ONBOARDING_DONE）
- 今日运营数据（当前状态、今日累计秒数等）
- 历史记录（HISTORY_RECORDS、WEEKLY_SESSION_RECORDS）

### 2.3 清除后行为
- 调用 `wx.reLaunch({ url: '/pages/onboarding/index' })` 进入引导页
- 用户需重新完成引导配置才能进入首页

## 三、每日 0 点重置逻辑

### 3.1 触发时机
每次小程序进入前台（`onShow`）或首页 `onLoad` 时：
将 `LAST_RESET_DATE`（YYYY-MM-DD）与当前系统日期对比，若不同则触发重置。

### 3.2 跨日边界处理（关键）
当日期变更发生在 App 挂起期间（后台或进程被杀）时：
- 当前仍在进行中的 Session（endTs=0）的 **跨日部分** 需要裁切：
  - 昨日贡献的秒数 = 昨日 23:59:59 时间戳 - startTs（上限为昨日的 endTs 边界）
  - 将这部分秒数归入昨日数据
  - 今日该段从今日 00:00:00 开始重新计时（或视为新 Session 开始于上班时刻）

### 3.3 重置流程（按顺序执行）
1. 打包今日数据为 `DailyRecord`，追加到 `HISTORY_RECORDS`（最多保留 90 条）
2. 清零今日累计字段：`TODAY_WORK_SECONDS`、`TODAY_FISH_SECONDS`、`TODAY_OVERTIME_SECONDS`
3. 将 `CURRENT_STATUS` 重置为 `'working'`，`STATUS_START_TIME = Date.now()`
4. 从 `WEEKLY_SESSION_RECORDS` 中删除 7 天前的历史记录
5. 更新 `LAST_RESET_DATE` 为今日日期

### 3.4 DailyRecord 格式

```
DailyRecord {
  date: string
  workSeconds: number
  fishSeconds: number
  overtimeSeconds: number
  workEarnings: number
  fishEarnings: number
}
```

## 四、今日工时秒数累计策略

### 4.1 核心思路
采用「已完成段累计值 + 当前进行段实时计算」的混合方案，**不每秒写 Storage**。

- `STATUS_START_TIME`：当前段开始时间戳（持久化于 Storage，App 重启后仍有效）
- `TODAY_WORK_SECONDS`：已完成的搬砖段秒数之和
- `TODAY_FISH_SECONDS`：已完成的摸鱼段秒数之和
- `TODAY_OVERTIME_SECONDS`：已完成的加班段秒数之和

### 4.2 状态切换时的存储操作
1. elapsed = (Date.now() - STATUS_START_TIME) / 1000
2. 将 elapsed 累加到对应 Key 并写 Storage
3. 将当前段标记为结束（更新 WEEKLY_SESSION_RECORDS 中对应 Session 的 endTs）
4. 创建新 Session 记录并追加到 WEEKLY_SESSION_RECORDS（endTs=0）
5. STATUS_START_TIME = Date.now()；CURRENT_STATUS = 新状态

### 4.3 实时展示（timer:second 回调，不写 Storage）
- 当前段已过秒数 = (Date.now() - STATUS_START_TIME) / 1000
- 今日摸鱼展示秒 = TODAY_FISH_SECONDS + 当前段秒数（仅 status=fishing 时加）
- 今日搬砖展示秒 = TODAY_WORK_SECONDS + 当前段秒数（仅 status=working 时加）
- 加班不计入收益，但加入在岗时长：TODAY_OVERTIME_SECONDS + 当前段秒数（仅 status=overtime 时加）

### 4.4 后台与切页时的行为
- **切换到后台或其他页面**：**不改变**任何状态，STATUS_START_TIME 保持当前值
- **定时器**：首页的 timer:second 在后台不运行，但这不影响数据——用户回到首页时，elapsed = (Date.now() - STATUS_START_TIME) / 1000 会自动包含后台期间的时间
- **重新进入首页**：在 onShow 中先检查是否跨日重置，再刷新展示数据

## 五、首页各数据项计算逻辑

### 5.1 倒计时

**场景 A：当前时刻 < 上班时刻（未到上班时间）**
- 标题：「距离上班还有」
- 显示：上班时刻 - 当前时刻，格式 HH:MM:SS（倒计时）

**场景 B：上班时刻 ≤ 当前时刻 ≤ 下班时刻（working 或 fishing 状态）**
- 标题：「逃离办公室倒计时」
- 显示：下班时刻 - 当前时刻，格式 HH:MM:SS（倒计时）
- 午休期间：倒计时照常继续，不暂停

**场景 C：当前时刻 > 下班时刻 + overtime 状态（加班中）**
- 标题：「今日无偿加班时长」
- 显示：当前时刻 - 下班时刻，格式 HH:MM:SS（正计时，越长越"血亏"）

**场景 D：当前时刻 > 下班时刻 + 非 overtime 状态（已下班未加班）**
- 显示固定文字：「已解放！」

---

### 5.2 今日已工位待命 X.X 小时

含义：今日「有效在岗总时长」（含加班，扣午休）

计算步骤：
1. 若当前时刻 < 上班时刻：返回 0.0
2. 上限时刻 = 若 overtime 状态：当前时刻；否则：min(当前时刻, 下班时刻)
3. 基础在岗分钟 = 上限时刻 - 上班时刻（分钟）
4. 若午休已开启：减去已过的午休分钟数 = min(max(当前时刻 - 午休开始时刻, 0), 午休总时长)
5. 有效在岗小时 = 有效在岗分钟 ÷ 60，保留一位小数

---

### 5.3 今日摸鱼收益 ¥X.XX

= 时薪 × 今日摸鱼展示秒数 ÷ 3600（见 4.3）

---

### 5.4 今日搬砖收益 ¥X.XX

= 时薪 × 今日搬砖展示秒数 ÷ 3600（见 4.3）

说明：
- **overtime 不计算收益**（视为无偿加班，让用户直观感受"白干"了多少）
- overtime 期间「搬砖收益」定格，不再增加
- overtime 时长单独展示（如「今日无偿加班时长」），作为情绪化信息强化

---

### 5.5 距发薪日 X 天

1. 读取 SALARY_DAY（1–31）
2. 若 SALARY_DAY 大于当月最大天数，取当月最后一天
3. 今日 = 发薪日当天：返回 0（今天到账）
4. 今日 < 发薪日：天数 = 发薪日 0 点 - 今日 0 点（向上取整天数）
5. 今日 > 发薪日：跨月计算，天数 = 下月发薪日 0 点 - 今日 0 点

---

### 5.6 距休息日 X 天

`day` = 今天星期几（0=周日，1=周一，…，6=周六）

**双休（double_rest）：**
- day ∈ {0, 6}：今天就是休息日，返回 0
- day ∈ {1–5}：返回 6 - day（距本周六）

**仅周六休（sat_only）：**
- day = 6：返回 0
- day = 0：下个周六还有 6 天，返回 6
- day ∈ {1–5}：返回 6 - day

**仅周日休（sun_only）：**
- day = 0：返回 0
- day ∈ {1–6}：返回 7 - day

**大小周（alternate）：**
1. 取参考周周一的 Date 对象（ALTERNATE_WEEK_REF_DATE）
2. 计算今日所在周周一与参考周周一相差的完整周数（整数）
3. 若周差为偶数：本周与参考周同类型；若奇数：本周为相反类型
4. 判断本周是大周还是小周后，按对应规则（双休或单休）计算距休息日天数

## 六、24 小时活动轨迹热力图

### 6.1 数据来源
从 `WEEKLY_SESSION_RECORDS` 中筛选出今日（或指定日期）的所有 SessionRecord，按时间排序。

### 6.2 时间轴分辨率
将 24 小时划分为 **48 格**（每格 30 分钟）。格索引 0 = 00:00–00:30，格索引 47 = 23:30–24:00。

### 6.3 每格着色算法
对每一格 `slot[i]`（时间区间 [start_i, end_i]）：
1. 遍历今日所有 SessionRecord，找出与该格有时间重叠的段
2. 计算每种类型（work / fish / overtime / rest）在该格中的覆盖分钟数
3. 取覆盖分钟数最多的类型作为该格颜色
4. 上班前和下班后无 Session 的格：标记为 rest

| 类型 | 颜色 |
|------|------|
| work | 蓝色（#2563eb，透明度按饱满度变化）|
| fish | 绿色（#10b981）|
| overtime | 橙红（#f97316）|
| rest | 浅灰（#e2e8f0）|

### 6.4 当前时刻标记
在对应的格上叠加一个「当前进度」指示器（如发光点或高亮边框）。

### 6.5 渲染方式
- WXML 中用 `wx:for` 遍历 48 个格子，每格为一个 `<view>`
- 每格的 `background-color`、`opacity` 通过 data 中的 `timelineSlots` 数组动态注入 style 属性
- 避免每秒重新渲染全部 48 格——仅当「状态切换」或「每整分钟」时触发热力图数据刷新（不在 timer:second 中每秒刷新）
